- **Strong story intelligence:** Later stories carried explicit “brownfield inventory” and read-before-change lists, which reduced accidental regressions when the repo already contained forward-looking stubs (mutations, routes).

- **Testing discipline:** Unit, integration, and E2E IDs from epics were referenced in story specs; MSW and Playwright patterns for load error and a11y gave repeatable signals for the read path.

## Challenges and system improvements

1. **Scope boundaries vs. reality**  
   Early scaffold and API work anticipated later epics (e.g. full CRUD surface). That was intentional as foundation, but **Story 1.4** still absorbed cross-cutting API/README/OpenAPI work that was not all in the written ACs. **Improvement:** Any change that touches **shared contracts** (CORS, Helmet, OpenAPI, global error handler) should be owned by the story whose ACs justify it, or split an explicit “platform” task so sprint scope stays legible.

2. **Validation stack semantics**  
   Fastify’s default AJV behavior (`removeAdditional`) interacted badly with strict PATCH expectations (extra `text` stripped before Zod). The failing test was correct; the stack behavior was surprising. **Improvement:** Treat “strict body shape” as an explicit architecture decision in Epic 2/3 stories (document chosen strategy: AJV config, pre-validation hook, or Zod-only body parsing).

3. **Integration tests and shared database**  
   Multiple stories noted **Vitest parallel workers + one `DATABASE_URL` + truncate** as a flake risk. **Improvement:** Before increasing parallel integration load, pick one strategy: serial integration project, `max: 1` pool, or isolated DB per worker.

4. **CI and local parity**  
   Playwright **global setup** running `docker compose` while CI also provides Postgres produced a **silent failure path** that still “works” by accident. **Improvement:** Guard Compose in CI or align setup with a single source of truth so failures are loud, not swallowed (tracked in `deferred-work.md`).

5. **Accessibility tooling consistency**  
   Unit path (`jest-axe`) vs E2E (`@axe-core/playwright`) can resolve different **axe-core** versions and diverge on violations. **Improvement:** Pin or align versions when a11y gates expand (Epic 3 / Story 3.4).
